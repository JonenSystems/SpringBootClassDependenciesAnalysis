基本画面要求一覧

■ 解析対象プロジェクト入力画面
REQ-101 要求 解析者がURI:/analysisにアクセスした際に解析対象プロジェクト入力画面を表示する
理由 - ユーザーが任意のプロジェクトを指定できるようにするため、入力フィールドおよび解析実行ボタンを提供する必要がある
説明
■ テキスト入力フィールドの提供
REQ-101.001 要求 解析者が解析対象プロジェクト入力画面を表示した際にルートパスを入力するためのテキスト入力フィールドを提供する
理由 - テキスト入力フィールドにより、任意のパスを柔軟に入力できる
説明 - 用語: ルートパスとは、解析対象となるJavaプロジェクトのルートディレクトリのフルパスを指す
- 用語: テキスト入力フィールドとは、HTMLのinput type="text"要素を指す
- 範囲: テキスト入力フィールドによるフルパス形式での入力を受け付ける機能を含む
- 例: C:\workspace\java\MyProject のような形式で入力される
- 例: <input type="text" name="projectPath" /> のような形式で実装される
■ テキスト入力フィールドの提供
SPC-101.001-001 [前提] 解析対象プロジェクト入力画面（URI:/analysis）が表示されていること。
[入力] 解析者が入力したルートパス（解析対象となるJavaプロジェクトのルートディレクトリのフルパス）を入力とする。
[処理] HTMLのinput type="text"要素を表示し、解析者がルートパスをテキスト入力できるようにする。入力値はバリデーションを実施し、有効なパス形式であることを確認する。
[出力] 入力されたルートパスを解析対象プロジェクト入力画面に表示し、解析実行ボタンがクリックされた際に値を受け渡し可能な状態にする。
[制約] 入力値はフルパス形式（例: C:\workspace\java\MyProject）である必要がある。相対パスや空文字列は無効とする。
■ 解析実行ボタンの提供
REQ-101.002 要求 解析者が解析対象プロジェクト入力画面を表示した際に解析実行ボタンを提供する
理由 - ユーザーが明示的に解析を開始できる操作要素を提供するため
説明 - 用語: 解析実行ボタンとは、HTMLのinput type="submit"要素を指す
- 範囲: ボタンの表示とクリックイベントの受付を含む
- 例: <input type="submit" value="解析実行" /> のような形式で実装される
■ 解析実行ボタンの提供
SPC-101.002-001 [前提] 解析対象プロジェクト入力画面（URI:/analysis）が表示されていること。
[入力] 解析者が解析実行ボタンをクリックしたイベントを入力とする。
[処理] HTMLのinput type="submit"要素を表示し、クリックイベントを受付ける。ボタンクリック時に、入力されたルートパスをリクエストパラメータとして送信し、解析処理を開始する。
[出力] 解析実行ボタンのクリックにより、解析処理が開始され、URI:/analysis/resultにリダイレクトされる。
[制約] ルートパスが入力されていない場合、または無効なパスの場合はボタンを無効化するか、エラーメッセージを表示する。
■ 解析結果画面
REQ-102 要求 解析者が解析対象プロジェクト入力画面の解析実行ボタンをクリックした後に、URI:/analysis/resultに遷移し、解析結果画面を表示する。
理由 - 解析処理の完了を解析者に伝え、解析結果のサマリから解析処理が正しく実施されたことを解析者が確認できるようにする
説明 - 用語: 解析結果とは、対象プロジェクトから抽出されたパッケージ、クラス、依存関係の情報を指す
- 範囲: 対象プロジェクトパス、パッケージリスト、クラス数、クラスリスト、依存関係の種類別件数リストの表示を含む
- 例: 解析実行後、画面にパッケージごとに整理されたクラスリストと依存関係が表示される
■ 対象プロジェクトパスの表示
REQ-102.001 要求 解析処理が完了した時に解析結果画面に解析対象となったプロジェクトのパスを表示する
理由 - どのプロジェクトが解析されたかを明確に示すことで、ユーザーが結果の対象を確認できる
説明 - 用語: 対象プロジェクトパスとは、解析処理が実行されたプロジェクトのルートディレクトリのフルパスを指す
- 範囲: 解析結果表示エリアにパス情報を表示する機能を含む
- 例: 「対象プロジェクト: C:\workspace\java\MyProject」のように表示される
■ 対象プロジェクトパスの表示
SPC-102.001-001 [前提] 解析処理が完了し、解析結果画面（URI:/analysis/result）が表示される状態であること。
[入力] 解析処理で使用されたプロジェクトのルートディレクトリのフルパスを入力とする。
[処理] 解析結果画面の表示エリアに、解析対象となったプロジェクトのパス情報を文字列として表示する。
[出力] 解析結果画面に「対象プロジェクト: [パス]」のような形式でプロジェクトパスを表示する。
[制約] パス情報は必ず表示され、空欄やnullは許容しない。表示形式は固定とする。
■ 解析結果の表示
REQ-102.002 要求 解析結果をパッケージごとに分割して表示する
理由 - プロジェクトに含まれる大量クラスを一度にリスト表示すると見にくいため、分割して表示する。
説明
■ 解析結果の表示
SPC-102.002-001 [前提] 解析処理が完了し、解析結果が取得済みであること。
[入力] 解析処理で取得された解析結果（パッケージ、クラス、依存関係の情報）を入力とする。
[処理] 解析結果をパッケージ単位でグループ化し、パッケージごとに分割して整理する。各パッケージの情報を独立したセクションとして画面に配置する。
[出力] 解析結果画面に、パッケージごとに分割された解析結果を表示する。
[制約] パッケージが存在しない場合は空のリストとして表示する。パッケージごとの表示順序は任意とする。
■ クラス名の重複チェック・除去・ソート機能
REQ-102.003 要求 解析結果（クラス名リスト）に対して重複を除去し、アルファベット順にソートした状態でサマリを作成する
理由 - 同じクラス名が複数回カウントされることを防ぐため、正確なクラス数を提示できる
説明 - 用語: 重複除去とは、クラス名リストから同一のクラス名を1つにまとめる処理を指す
- 範囲: クラス名の重複チェック、重複除去、重複除去後のカウントを含む
- 例: 「UserController, ProductController, UserController」というリストから「ProductController, UserController」に変換し、クラス数は2とカウントする
■ クラス名の重複チェック・除去・ソート機能
SPC-102.003-001 [前提] 解析結果が取得済みで、クラス名リストがパッケージごとに整理された状態であること。
[入力] 解析結果から取得したクラス名リスト（同一パッケージ内で重複を含む可能性があるクラス名のコレクション）を入力とする。
[処理] Javaの`Stream.distinct()`メソッドを使用してクラス名の重複を除去し、`Set<Class>`を使用してエンティティレベルでの重複も防止する。重複除去後、クラス名をアルファベット順にソートして表示用のリストを作成する。クラス数のカウントも重複除去後の数で計算する。`Class`エンティティの`equals`と`hashCode`は`fullQualifiedName`に基づいて実装する。
[出力] 重複が除去され、アルファベット順にソートされたクラス名リストを解析結果画面に表示し、パッケージごとのクラス数も重複除去後の数で表示する。
[制約] `Class`エンティティの`fullQualifiedName`が正しく設定されていることが前提である。同一パッケージ内での重複のみを対象とし、異なるパッケージ間での同名クラスは別クラスとして扱う。
■ パッケージ名の表示
REQ-102.004 要求 各パッケージのパッケージ名をパッケージごとに整理された状態で解析結果画面に表示する
理由 - 取得されたパッケージ名よりから解析処理が正しく実施されたことを解析者が確認できるようにする
説明 - 例: 「com.example.controller」「com.example.service」「com.example.repository」のように表示される
■ パッケージ名の表示
SPC-102.004-001 [前提] 解析処理が完了し、解析結果がパッケージごとにグループ化された状態であること。
[入力] 各パッケージのパッケージ名（完全修飾名）を入力とする。
[処理] パッケージごとに整理された解析結果から、各パッケージのパッケージ名を抽出し、パッケージごとのセクションに表示する。
[出力] 解析結果画面に、パッケージごとに整理された状態で各パッケージ名を表示する。
[制約] パッケージ名は完全修飾名（例: com.example.controller）として表示する。パッケージ名が存在しない場合は表示しない。
■ クラス数の表示
REQ-102.005 要求 各パッケージ含まれるクラス数をパッケージごとに整理された状態で解析結果画面に表示する
理由 - 検出されたクラス数から解析処理が正しく実施されたことを解析者が確認できるようにする
説明 - 用語: クラス数とは、各パッケージ内に存在するユニークなクラス名の数を指す
- 範囲: パッケージごとのクラス数の集計と表示機能を含む
- 例: 「検出されたクラス数: 5」のように表示される
■ クラス数の表示
SPC-102.005-001 [前提] 解析処理が完了し、各パッケージに含まれるクラス情報が取得済みであること。クラス名の重複チェック・除去処理が完了していること。
[入力] 各パッケージに含まれるクラス情報（重複除去後のクラスリスト）を入力とする。
[処理] 各パッケージに含まれるクラス数をカウントし、パッケージごとに整理された状態で数値を算出する。
[出力] 解析結果画面に、パッケージごとに整理された状態で各パッケージのクラス数を表示する。
[制約] クラス数は重複除去後のユニークなクラス名の数でカウントする。クラスが存在しないパッケージは0件として表示する。
■ クラスリストの表示（パッケージごと、重複除去・ソート後）
REQ-102.006 要求 各パッケージに含まれるクラスリストを、パッケージごとにアルファベット順に整理された状態で解析結果画面に表示する
理由 - パッケージごとのクラス一覧を表示することで、解析処理が正しく実施されたことを解析者が確認できるようにする
説明 - 用語: クラスリストとは、各パッケージ内に存在するクラス名の一覧を指す（重複除去後、アルファベット順ソート済み）
- 範囲: パッケージごとのクラス名の一覧表示機能を含む
- 例: 「com.example.controller: UserController, ProductController, OrderController」のように表示される
■ クラスリストの表示（パッケージごと、重複除去・ソート後）
SPC-102.006-001 [前提] 解析処理が完了し、各パッケージに含まれるクラス情報が取得済みであること。クラス名の重複チェック・除去・ソート処理が完了していること。
[入力] 各パッケージに含まれるクラスリスト（重複除去済み、アルファベット順ソート済み）を入力とする。
[処理] パッケージごとに整理されたクラスリストを、アルファベット順にソートされた状態で表示用に整形する。
[出力] 解析結果画面に、パッケージごとにアルファベット順に整理された状態でクラスリストを表示する。
[制約] クラス名は重複除去後、アルファベット順にソートされた状態で表示する。クラスが存在しないパッケージは空のリストとして表示する。
■ 依存関係の種類別件数リストの表示
REQ-102.007 要求 各パッケージに含まれる依存関係の種類別件数をパッケージごとに整理された状態で解析結果画面に表示する
理由 - 依存関係の種類と件数を把握することで、解析処理が正しく実施されたことを解析者が確認できるようにする
説明 - 用語: 依存関係の種類とは、型依存、メソッド依存、フィールド依存などの依存関係のカテゴリを指す
- 範囲: パッケージごとの依存関係の種類別集計と表示機能を含む
- 例: 「com.example.controller: 型依存 10件、メソッド依存 5件、フィールド依存 3件」のように表示される
■ 依存関係の種類別件数リストの表示
SPC-102.007-001 [前提] 解析処理が完了し、各パッケージに含まれる依存関係情報が取得済みであること。
[入力] 各パッケージに含まれる依存関係情報（種類別の依存関係レコード）を入力とする。
[処理] 各パッケージに含まれる依存関係を種類（型依存、メソッド依存、フィールド依存など）ごとに集計し、パッケージごとに整理する。
[出力] 解析結果画面に、パッケージごとに整理された状態で各パッケージの依存関係の種類別件数を表示する。
[制約] 依存関係の種類は定義されたカテゴリに基づいて分類する。依存関係が存在しない種類は0件として表示する。
